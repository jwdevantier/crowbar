CROWBAR

"When clever hacking fails, crude whacking prevails!"

# Abstract

Crowbar is a code- and content-generation tool. It allows you to embed blocks of Python inside a file to dynamically generate content inside an otherwise static file.

# What does it do?

Crowbar principally works as a preprocessor. It scans your file for specially formatted blocks, evaluates the code in those blocks and inserts the output of those evaluations back into the source file.

For example, say we wanted to generate some forward declarations of some C functions:

// This is my C file
/*<<crowbar
  for fname in ["do_something", "do_another_thing", "do_last_thing"]:
      emit(f"void mylib_{fname}();")
  >> */
//<<end>>

After running, the file would look like so:

// This is my C file
/*<<crowbar
  for fname in ["do_something", "do_another_thing", "do_last_thing"]:
      emit(f"void mylib_{fname}();")
  >> */
void mylib_do_something();
void mylib_do_another_thing();
void mylib_do_last_thing();
//<<end>>

Lines with <<crowbar, >> and <<end>> tokens are marker lines.

Important; each of these tokens must be aligned in the block - this is required to not confuse contents in the code- and output for a legitimate marker line.

# NOTE: in this example, we use Python-style line comments.
# The `#` prefixed every marker line here has nothing to do with Crowbar!

# <<crowbar
  ... python code here ...
# >>
... output here ...
# <<end>>

Crowbar works for any language or file format which can support line- *or* multi-line comments in some way. Because of this, you have to indent each line of code in your blocks to match the start of the <<crowbar marker, as shown above.

Also, if you want to write the output elsewhere, you can also omit the code-blocks themselves from the output. This is helpful when writing small one-shot/passive generators, as you would for project starter templates and such. To omit the blocks from the output, run crowbar like so: 

python crowbar.py --no-code-blocks input-file output-file

# The emit() API

emit()is the equivalent of print() for Crowbar. Each argument is a Component, a marker tokenor a string(ifiable) value. By default, each argument is placed on a separate line, unless the line continue (lc) token is used to continue output on the same line.

Components are annotated functions, which act as composable "templates" or building-blocks for generating more complex output. We cover those later.

Marker tokens are used to control newlines and indentation, the marker tokens are:

  • indent – increase level of indentation used for new lines
  • dedent – decrease level of indentation used for new lines
  • lc (line-continue) – each parameter to emit is normally placed on its own line, this ensures the next element follows directly after the former, on the same line.
  • nl (newline) – insert (additional) newline at this point
  • fl (fresh line) – insert newline iff. line already has content on it (only relevant if you want to override a lc token)

This API makes formatting output much easier. In particular, abstracting indentation out this way permits components to be composable blocks, which may be nested in various ways and yet still the combined output is properly indented.

# Using lists to represent indentation

Crowbar strives to handle complex output much better than templating, but still make it easy to recognize the shape of the generated output. To that end, you can use python lists to wrap elements that must be indented. This makes it easier to see in the code how the generated output uses indentation, for example:

# <<crowbar
# emit(
# 'if condition:', [
#    'print("then this")',
#    'if something_more:', [
#        'print("inner condition")'
#    ],
#    'print("still in outermost if-block")'
# ],
# 'print("outside if-blocks")'
# )
# >>
if condition:
    print("then this")
    if something_more:
        print("inner condition")
    print("still in outermost if-block")
print("outside if-blocks")
# <<end>>
# Components - compose and reuse!

Components is an additional feature that really separates Crowbar from tools like Cog. Think of components as building blocks, responsible for rendering some smaller element, and which can in turn delegate to other components. A component takes an emit() function, but can take any additional arguments (positional and by keyword). When the component is invoked, the resulting value can be passed to emit() or as a parameter to another component. Either way, indentation will just work™

# Components Example

Below we write a series of components which collectively make rendering a C function easy. Because we want to define several components, we put them in a separate Python file. This means we get full editor support: syntax highlighting, LSP, debugging and more.

While this is a lot of code to print out one C function, it can pay off when you have a larger list of entities to render out.

from crowbar import *

@component
def join(emit, sep=", ", *components):
    """Insert `sep` between each element."""
    if len(components) == 0:
        return
    if not isinstance(sep, list):
        sep = [sep]
    emit(components[0])
    for c in components[1:]:
        emit(*sep, c)

@component
def cfunc(emit, label=None, args=None, ret="void", body=None):
    if args is None:
        args = []
    if label is None:
        raise RuntimeError("must name the function")
    emit(
        f"{ret} {label}(",
        # use `lc` place elements on the same line
        lc, join([lc, ", ", lc], *args),
        lc, ") {",
        # alternatively; wrap elements to be indented in a list
        # [ body ]
        indent, body, dedent,
        "}"
    )

@component
def param(emit, ctype, label):
    emit(f"{ctype} {label}")

@component
def zebody(emit):
    emit(f"return x + y;")

We can then generate a C function using a block like this:

/*<<crowbar
  from c_func_utils import *
  emit(
      cfunc(
          label="add",
          args=[param("int", "x"), param("int", "y")],
          ret="int",
          body=zebody(),
      ),
  )
  >> */
int add(int x, int y) {
  return x + y;
}
//<<end>>
# Infrequently asked questions
# How does crowbar find blocks?

Crowbar only looks for those "marker lines", lines containing <<crowbar, >> and <<end>>.

# How does crowbar extract code from a block?

It takes every line between the line containing <<crowbar and the line containing >>, then strips off the first N characters from each line, where N is the number of characters between the start of the line and the first `<` in <<crowbar

This is why we say that Crowbar is language agnostic, this approach, while imposing stricter requirements on indenting code lines, allows us to use Crowbar in any type of file so long as that syntax supports line or multi-line comments.

# How are blocks processed?

Each file is read once, top to bottom, and each block is parsed in-order. If any block raises an exception, processing stops and the file remains unchanged. Variables defined in one block are available to subsequent blocks.

# How do I configure indentation?

A line's indentation is essentially N times some indentation string, where N corresponds to the indentation level.

If using crowbar as a preprocessor (python crowbar.py <file>), add the `--indent-step '<string>'` flag to specify the indentation string.

If using Crowbar's API from a Python script, pass `indent_step="<string>"` when creating an Emitter.

# Does my indentation of the block affect the indentation of the output?

No. You only indent the block as you want to make it look nice within the file. The only things affecting how the output is indented are:

  • The current indentation level, controlled by indent and dedent marker tokens.
  • The chosen indent_step string. See the question above.
